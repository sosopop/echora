# Task Handoff · AI 协作交接约定

> **本约定为永久工作流,适用于所有未来任务。每次会话开始前请先读本文。**

## 规则

每次涉及代码或文档变更的任务,执行结束时必须产出两件事:

### 1. `doc/task/<NNN>-<slug>.md` — 任务执行记录

固定四段结构:
- `## 任务背景` — 一两句话说明这次做了什么、起因
- `## 执行摘要` — 实际产出的关键变更(文件清单 + 简述,不复述每行代码)
- `## 遗留 TODO` — 计划中未完成、被显式延后或新发现的项,带责任域(后端/前端/文档/测试)
- `## 下一阶段建议` — 基于 `doc/prd.md` 推断的合理下一步,3-5 条,每条注明 PRD 章节定位与价值理由

### 2. 同步维护本文件

若本约定本身被调整(例如新增段落、修改文件名规则),需同步更新本文件并在 commit 信息中说明。

## 命名与存放

- 目录:`doc/task/`(注意与 `doc/milestone/` 区分:milestone 是前瞻里程碑计划,task 是回顾执行记录)
- 文件名:**`<NNN>-<kebab-slug>.md`**,其中 `NNN` 是 0 填充的 3 位序号,从 `001` 起递增
  - 示例:`001-foundation-framework.md` → `002-onboarding-flow.md` → `003-scene-select-skill.md`
  - 序号在文件名;**任务执行日期写入文件首行 `> 日期: YYYY-MM-DD`**,与序号解耦
- 一次任务一个文件,不追加;同一任务跨日继续也复用原序号文件,只更新首行日期范围(如 `> 日期: 2026-05-16 ~ 2026-05-18`)
- 创建新文件前先 `ls doc/task/` 取最大序号 +1
- 即使任务被废弃也保留序号占位,避免歧义

## 与其他文档的关系

- `doc/prd.md` — 产品需求事实源,下一阶段建议必须引其章节
- `doc/esd.md` — 工程规范事实源,执行约束与 commit 前缀均按 ESD 走
- `doc/milestone/` — 前瞻里程碑计划(尚未落地),与 task/ 是镜像关系(milestone 描述要做什么,task 记录做了什么)
- `doc/knowledge/architecture.md` 等主题文档 — 任务中如果引入新的架构决策,需同步更新对应主题文档

## 触发场景

- 任何新增功能、修复 bug、重构、文档大改的会话结束时
- 不含范围:仅询问 / 仅阅读 / 仅运行命令的会话不需要产出 task 记录

