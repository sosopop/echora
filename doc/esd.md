# 通用技术栈与项目管理规范

本文档抽取当前仓库中可复用的技术栈、工程结构和协作规范，用于复刻到其他全栈 TypeScript 项目。文档只保留通用工程实践，不包含具体产品、业务领域、页面功能或数据特性的规则。

## 1. 技术栈基线

### 1.1 运行时与语言

- Node.js + TypeScript 作为全栈基础。
- 前后端均开启 TypeScript `strict`，优先用类型系统约束跨层契约。
- 项目使用 ESM：`package.json` 设置 `"type": "module"`。
- 服务端 TypeScript 使用 `NodeNext` / `NodeNext`，本地模块导入保持 `.js` 扩展名。
- 前端 TypeScript 使用 Vite Bundler 解析：`moduleResolution: "Bundler"`。

### 1.2 前端

- React 18 作为 UI 框架。
- Vite 作为开发服务器、构建工具和前端测试配置入口。
- Zustand 管理全局状态。
- React Router 管理页面路由。
- Vitest + jsdom + React Testing Library 覆盖前端单元测试和组件测试。
- 图表、图标、字体、动效类依赖应作为 UI 层能力接入，避免把具体依赖泄漏到业务无关的共享层。

### 1.3 后端

- Express 作为 HTTP 服务框架。
- SQLite 作为本地持久化数据库，使用 `better-sqlite3` 访问。
- SQL 迁移文件独立放置，由启动或命令显式执行迁移。
- `dotenv` 读取本地环境变量。
- JWT 处理登录态或 API 鉴权令牌。
- `bcryptjs` 处理密码或敏感凭据哈希。
- `zod` 可用于运行时输入校验和边界数据解析。
- `cors`、`multer`、HTTPS/证书相关依赖按项目需要接入，但应保持在服务边界或基础设施层。

### 1.4 测试与工具

- Jest + ts-jest + supertest：后端路由、服务和配置测试。
- Vitest + jsdom：前端 store、组件、视图和工具函数测试。
- tsx：直接运行 TypeScript 脚本、开发期 watch 和烟雾测试入口。
- TypeScript 编译与 Vite 构建分离，便于分别定位服务端和前端问题。

## 2. 推荐目录结构

```text
.
├── server/              # 后端源码：启动、路由、服务、数据库、配置
├── src/                 # 前端源码：入口、路由、状态、视图、组件、API client
├── shared/              # 前后端共享类型、常量、纯函数和协议定义
├── migrations/          # 数据库迁移 SQL
├── tests/smoke/         # 跨层烟雾测试或端到端串联脚本
├── server/__tests__/    # 后端测试
├── src/__tests__/       # 前端测试
├── data/                # 运行时数据目录，按项目需要决定是否纳入仓库
├── db/                  # 本地数据库目录，通常不作为源码维护
├── dist-server/         # 后端构建产物
├── dist-web/            # 前端构建产物
├── release/             # 发布产物目录
└── doc/                 # 产品、工程、知识库和项目管理文档
```

目录边界原则：

- `server/`、`src/`、`shared/`、`migrations/`、测试目录是主要事实源。
- `dist-server/`、`dist-web/`、`release/` 是构建产物，不作为源码维护。
- `shared/` 只放跨层稳定契约，不放依赖浏览器、数据库或框架实例的代码。
- `doc/` 下的知识库负责索引、流程、约束和排障路径；实现细节以源码和测试为准。

## 3. 配置规范

### 3.1 配置来源优先级

推荐统一配置优先级：

```text
环境变量 > 配置文件 > 代码默认值
```

落地建议：

- 服务端配置集中由 `getConfig()` 一类函数读取和归一化。
- 支持 `SERVER_CONFIG_PATH` 指定配置文件路径。
- 配置文件允许 `UPPER_SNAKE_CASE` 和 `camelCase` 两种键名，方便部署与本地开发兼容。
- 路径类配置在读取后解析为绝对路径，避免工作目录变化导致运行时漂移。
- 布尔值、数字、数组等配置必须在边界处归一化。
- 密钥、令牌、证书路径等敏感信息不得写入通用文档或提交到仓库。

### 3.2 前端环境变量

- Vite 前端只读取 `VITE_` 前缀环境变量。
- API 基地址、公开开关、客户端可见配置必须显式使用 `VITE_` 前缀。
- 不要假设普通服务端环境变量会自动暴露给前端。

## 4. 启动与构建命令

推荐保留以下脚本入口：

```json
{
  "scripts": {
    "dev": "node scripts/dev-server.js",
    "dev:web": "vite",
    "build": "tsc -p tsconfig.server.json && vite build",
    "test": "npm run test:server && npm run test:web && npm run test:smoke",
    "test:unit": "npm run test:server && npm run test:web",
    "test:server": "node --experimental-vm-modules ./node_modules/jest/bin/jest.js --runInBand",
    "test:web": "vitest run",
    "test:smoke": "tsx tests/smoke/run-smoke.ts",
    "migrate": "tsx server/db/migrate.ts",
    "release": "node scripts/release.js"
  }
}
```

命令语义：

- `dev`：只启动后端热更新，并显式设置 `NODE_ENV=development`。
- `dev:web`：只启动前端开发服务器；全栈联调时与后端服务分开启动。
- `build`：先编译后端，再构建前端。
- `test:unit`：最小单元门禁，适合大多数代码变更后的快速验证。
- `test`：完整回归，包含后端、前端和烟雾测试。
- `migrate`：显式执行数据库迁移。
- `release`：生成发布目录，只复制运行所需产物和配置。

如果仓库没有 `lint` 或 `format` 脚本，不要在文档、CI 或交付说明中假设它们存在。

## 5. 后端工程规范

### 5.1 启动链路

推荐启动链路保持清晰、可测试：

```text
读取配置 -> 创建数据库连接 -> 执行迁移 -> 装配应用 -> 启动 HTTP/HTTPS 服务
```

原则：

- 启动文件只负责组装，不承载业务逻辑。
- `createApp()` 返回 Express app，便于测试中直接注入 supertest。
- 数据库连接和配置对象由入口注入，避免模块加载时产生隐式副作用。
- 迁移应可在启动链路和独立命令中复用。

### 5.2 路由、服务与数据层

- 路由层负责 HTTP 输入输出、鉴权上下文和错误映射。
- 服务层负责业务编排、事务边界和跨模块规则。
- 数据访问层负责 SQL、持久化结构和查询聚合。
- 共享类型放入 `shared/`，避免前后端重复声明 API 契约。
- 所有外部输入在进入核心逻辑前完成校验、归一化和错误分类。
- 错误响应应保持稳定格式，便于前端统一处理。

### 5.3 数据库与迁移

- 数据库结构变更必须新增迁移，不直接依赖手工改库。
- 迁移脚本应记录已执行版本，重复运行必须安全。
- 测试环境使用隔离数据库或临时数据库，避免污染开发数据。
- 构建产物和发布目录不应被视为数据库事实源。

## 6. 前端工程规范

### 6.1 应用启动

推荐启动顺序：

```text
创建 React app -> 初始化 Zustand stores -> 配置 API client -> 恢复会话/初始化状态 -> 注册 Router -> 挂载应用
```

原则：

- API client 集中处理基地址、鉴权令牌和 401 回调。
- 全局状态恢复在路由挂载前完成，避免首屏权限状态闪烁。
- 路由、store、API client 之间通过明确回调或组合函数协作，不互相隐式导入运行时实例。

### 6.2 状态与组件

- Zustand store 承载跨页面状态、异步动作和缓存。
- 组件优先保持展示和交互职责，复杂流程下沉到 store、自定义 hooks 或工具模块。
- API 返回值先在边界层归一化，再进入 store。
- 可复用 UI 组件不应直接依赖具体页面流程。
- 页面测试覆盖关键状态转换、API 错误分支和用户可见结果。

## 7. 测试策略

### 7.1 测试分层

- 后端测试：覆盖路由、服务、配置解析、数据库迁移和错误映射。
- 前端测试：覆盖 store、视图、组件、路由协作和 API client 行为。
- 烟雾测试：覆盖从启动服务到关键 HTTP/API 流程的跨层串联。

### 7.2 最小测试选择

- 改 `server/routes/*`、`server/services/*`、`server/db/*`：至少跑 `npm run test:server`。
- 改 `src/stores/*`、`src/views/*`、`src/components/*`、`src/api/*`：至少跑 `npm run test:web`。
- 改 `shared/*`、配置优先级、鉴权、跨层 API 契约、发布路径或迁移：优先跑 `npm test`。
- 只改文档：至少检查链接、文件路径和是否误写实现细节。

### 7.3 失败处理

- 用户或 CI 提供失败日志时，先用当前代码复现，不直接假设日志仍然有效。
- 先区分环境、数据、配置、运行方式问题，再判断是否为代码回归。
- 修复后补最小覆盖测试；如果无法自动化，记录手工验证步骤。

## 8. 发布规范

- 发布脚本应先执行构建，再生成干净的 `release/` 目录。
- 发布目录只包含运行所需的后端产物、前端静态文件、迁移文件、生产 `package.json` 和启动脚本。
- 运行时数据、数据库、证书和私密配置是否复制，应明确写入部署说明，不默认混入发布产物。
- 生产 `package.json` 只保留运行依赖和启动命令。
- 启动脚本负责设置生产环境默认值，但不得覆盖显式传入的环境变量或配置文件。

## 9. 文档与知识库规范

### 9.1 文档分层

推荐把 `doc/` 分为：

- `prd/`：产品背景和需求说明，不作为实现事实源。
- `knowledge/`：面向开发和 AI 协作的工程知识库。
- `task/`：具体任务、方案、阶段计划或临时交付材料。
- 通用规范文档：技术栈、协作流程、发布规范、测试策略等。

### 9.2 知识库写法

- 根索引只做入口和问题路由，不承载长正文。
- 单篇文档只覆盖一个主题或一条链路。
- 正文优先写修改入口、约束、依赖、失败点和测试入口。
- 引用源码时写文件名与函数名/类名，不依赖易漂移的行号。
- 暂时没把握的结论放入 pending 区域，确认后再迁入正式文档。
- 任何公共行为、配置优先级、API 契约、共享类型、测试入口、构建和发布路径变更后，都要同步检查知识库。

## 10. 项目管理流程

### 10.1 需求进入

- 新需求先落成短文档，明确目标、非目标、用户可见变化、数据影响、接口影响和验收方式。
- 不确定的问题单独列为待确认项，不把猜测写成确定规范。
- 如果需求涉及公共行为、配置、权限、数据结构或发布路径，必须在方案阶段标明影响面。
- 需求文档描述“要解决什么”和“验收什么”，避免提前固化实现细节。

### 10.2 任务拆解

- 将任务拆成可独立验证的工程切片，例如后端契约、前端状态、数据库迁移、测试、文档。
- 每个切片都要有清晰完成条件和推荐测试入口。
- 跨层改动优先从共享类型或 API 契约开始，再分别实现后端和前端。
- 大任务保留实施计划，记录执行顺序、风险点和回滚方式。

### 10.3 里程碑管理

- 里程碑文档只描述阶段目标、交付边界、验收标准和剩余风险。
- 已完成、进行中、待确认、延期项要分开记录。
- 里程碑完成后，根据真实实现反向更新知识库，而不是只更新计划文档。
- 发布前确认构建产物、运行时数据、配置文件、迁移和部署步骤是否都被覆盖。

### 10.4 变更控制

- 修改公共契约时，同步检查调用方、测试、文档和发布说明。
- 修改配置优先级、默认路径、启动脚本或发布目录时，必须视为基础设施变更处理。
- 数据库结构变更必须同时包含迁移、测试或手工验证说明。
- 临时方案要标注有效期或后续清理条件，避免沉淀成隐性架构。

### 10.5 验收与交付

- 交付说明包括变更摘要、验证命令、未验证原因和残余风险。
- 文档类任务至少确认路径、标题、适用范围和是否混入项目特定内容。
- 代码类任务至少确认测试结果、配置影响、迁移影响和发布影响。
- 如果无法完成自动化验证，必须记录手工验证步骤和限制。

## 11. Git 与协作规范

### 11.1 提交前检查

提交前至少确认：

- 变更范围和任务目标一致，没有混入无关重构。
- 未修改构建产物，除非任务明确要求更新发布包。
- 配置、迁移、测试和文档是否需要同步更新。
- 已运行与变更范围匹配的最小测试。
- 没有提交本地密钥、真实证书、数据库文件或私有运行时数据。

### 11.2 Commit 前缀

使用固定前缀保持历史可检索：

- `feat: 描述`：新增能力。
- `fix: 描述`：缺陷修复。
- `refactor: 描述`：不改变外部行为的重构。
- `docs: 描述`：文档变更。
- `test: 描述`：测试补充或测试基础设施变更。

### 11.3 代码评审重点

评审优先看：

- 是否改变外部行为、配置优先级或 API 契约。
- 是否破坏前后端共享类型。
- 是否绕过统一配置、鉴权、错误处理或 API client。
- 是否缺少数据库迁移或迁移不可重复执行。
- 是否缺少与风险匹配的测试。
- 是否把项目特定知识写进了通用规范。

## 12. 新项目复刻清单

创建新项目时，可按以下顺序落地：

1. 初始化 Node.js + TypeScript + ESM。
2. 建立 `server/`、`src/`、`shared/`、`migrations/`、`tests/smoke/` 和 `doc/`。
3. 配置服务端 `tsconfig.server.json`，使用 `NodeNext` 和 `strict`。
4. 配置前端 `tsconfig.json`、Vite、React、Zustand、React Router。
5. 建立统一 `getConfig()`，落实环境变量、配置文件、默认值三层优先级。
6. 建立后端启动链路：配置、数据库、迁移、app 装配、服务启动。
7. 建立 API client，集中处理基地址、鉴权令牌和 401。
8. 配置 Jest、Vitest、smoke 三层测试入口。
9. 建立构建和发布脚本，区分源码、构建产物、发布产物和运行时数据。
10. 建立 `doc/knowledge/README.md`，写清源码事实源、测试入口和知识库维护规则。
11. 约定 commit 前缀、测试门禁和 PR 检查清单。
12. 首个功能完成后，用真实改动反向校验目录、配置、测试和文档是否顺手。
