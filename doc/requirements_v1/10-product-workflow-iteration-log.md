# 10. 产品工作流发散式迭代记录

> 本文是需求层的持续审计记录，不是最终定稿。每发现一个问题，都可以长出多个方向：有的进入 MVP，有的保留为候选，有的暂不做，有的明确砍掉。当前可执行实现基准仍以 `08-ai-workflow-and-widget-contract.md` 为准。

## 1. 使用方式

### 1.1 记录原则

- 每个问题都是一次迭代条目，不要求一次性收敛完。
- 每个迭代条目允许同时产生多个目标方向，包括 MVP 修正、体验增强、技术约束、后续版本和砍掉项。
- 发散想法必须有状态，不能悬空留在需求里。
- 标记为 `MVP 保留` 或 `MVP 收敛` 的内容，只有同步回 `08` 契约后才算进入实现基准。
- `demo.html` 只展示当前 MVP 选定路径；`workflows.html` 可以展示候选分支和取舍理由。

### 1.2 状态标签

| 状态 | 含义 | 是否进入 demo 主路径 |
|------|------|----------------------|
| `MVP 保留` | 当前版本必须实现，直接服务主学习流 | 是 |
| `MVP 收敛` | 想法有价值，但当前版本只保留最小形态 | 是，只展示收敛后的形态 |
| `后续候选` | 有产品潜力，但不作为 V1 依赖 | 否，只在工作流说明中标注 |
| `暂不做` | 当前阶段不实现，避免范围膨胀 | 否 |
| `已砍掉` | 与主线冲突或无法可靠落地 | 否 |

## 2. 当前产品树基线

### 2.1 根

Echora V1 的根是“AI 会话流式学习”：用户不在多个页面之间搬运学习任务，而是在一条连续学习流里完成场景选择、练习、批改、追问、复盘和重练。

### 2.2 主干

- 单一 active 主学习流。
- 中间对话内容列表承载学习数据流。
- 底部交互区承载所有主线输入和学习菜单。
- Widget 是对话中的功能小部件，不是独立页面。
- AI 负责生成、解释、推荐和调度建议；系统负责确定性动作和状态安全。

### 2.3 一级分支

| 分支 | 当前 MVP 选择 | 风险控制 |
|------|---------------|----------|
| 账号入口 | 作为 `account-gate` Widget 在保存或注册时出现 | 真实鉴权由系统执行，不交给 AI |
| 历史会话 | 左侧列表展示当前、已完成、受限会话 | 练习中隐藏答案和批改详情 |
| 场景选择 | `scene-cards` + 自定义场景输入 | 自定义描述仍需系统确认后进入练习 |
| 练习答题 | `exercise-card`、`fill-blank`、`choice-question` | 答案提交必须写入结构化 attempt |
| 批改反馈 | `grading-result` 写入分数、错因、错误标签 | 复盘不从自然语言临时解析 |
| 辅助追问 | 右侧支线解释来源上下文 | 不改变主线状态，不直接生成下一题 |
| 复盘重练 | `progress-summary` + `answer-review` + 重练入口 | 重练题基于结构化记录生成 |

## 3. 迭代条目

### ITER-001：主学习流会不会被做成多页面学习平台

**问题**

旧方案容易自然滑回 Dashboard、错题本、设置页、复盘页等多页面结构。这样虽然熟悉，但会削弱“对话即界面”的核心特色，也会让用户在学习过程中不断跳出主线。

**发散想法**

- `MVP 保留`：所有主线学习动作只在中间学习流和底部交互区发生。
- `MVP 收敛`：复盘、错题、薄弱点只作为学习流中的 Widget 和摘要出现。
- `后续候选`：如果后续用户需要长周期管理，可增加独立学习档案页，但只能作为历史分析，不承接正在练习的主线。
- `已砍掉`：V1 不恢复旧式 Dashboard / 错题本 / 设置页作为主导航结构。

**MVP 决策**

保留单主学习流。任何功能如果不能自然嵌入消息流或底部交互区，就不进入 V1 MVP。

**原型映射**

`demo.html` 只展示三栏工作台；`widgets.html` 只展示 Widget 契约；`workflows.html` 标注旧式多页面结构为已砍掉。

### ITER-002：学习菜单会不会变成隐藏命令系统

**问题**

底层需要结构化 action，但用户不应该理解或输入命令。如果界面出现命令感，会破坏自然对话体验，也会让学习产品显得像开发者工具。

**发散想法**

- `MVP 保留`：输入框左侧使用普通学习菜单按钮，菜单项是自然语言。
- `MVP 收敛`：菜单只保留开始练习、查看复盘、复习薄弱点、换个场景、保存进度。
- `后续候选`：根据用户阶段动态排序菜单项。
- `暂不做`：不支持用户手输快捷指令。
- `已砍掉`：不在用户界面展示内部 action、路由、置信度或调试事件。

**MVP 决策**

菜单是产品功能入口，不是命令入口。点击菜单后系统生成结构化 action，先校验状态，再决定是否执行、确认或交给 AI 生成内容。

**原型映射**

`demo.html` 展示普通学习菜单；`workflows.html` 展示 action matrix，但只作为需求解释，不出现在真实产品体验里。

### ITER-003：Widget 抽象会不会过宽，导致所有功能都想变成 Widget

**问题**

“所有功能都可以由 AI 调用 Widget”容易被误解为 Widget 可以承接任何复杂业务。若边界不清，登录、保存、提交、鉴权等确定性动作会被错误交给 AI。

**发散想法**

- `MVP 保留`：V1 固定 12 个 Widget，全部服务主学习流。
- `MVP 收敛`：登录注册作为 `account-gate`，是账号入口 Widget，不是真实认证逻辑本身。
- `后续候选`：新增发音评测、语音跟读、长文阅读等 Widget，需要单独定义契约。
- `暂不做`：不做通用表单构建器，不让 AI 任意拼装未知 Widget。
- `已砍掉`：不允许 AI 直接执行系统权限动作。

**MVP 决策**

Widget 负责展示和收集用户意图，系统负责安全执行。新增 Widget 必须先进入契约清单，再进入实现。

**原型映射**

`widgets.html` 只展示 12 个 V1 Widget，不展示候选 Widget；`workflows.html` 标注候选扩展方向。

### ITER-004：辅助追问如何有价值但不打断主线

**问题**

用户在批改或题目中间追问很有价值，但普通聊天会把当前流程带偏。若追问可以直接生成下一题或切换场景，主线状态会变得不可预测。

**发散想法**

- `MVP 保留`：辅助追问只能围绕 `sourceRef` 解释、举例、改写。
- `MVP 收敛`：辅助追问默认不写入薄弱点统计，除非用户明确加入复盘。
- `后续候选`：允许用户把某条追问总结插回主线，形成新的练习方向。
- `暂不做`：不在支线里直接生成正式下一题。
- `已砍掉`：不允许支线改变主学习流状态。

**MVP 决策**

辅助追问是“临时支线”，不是第二条主线。它可以延展理解，但不能替代主学习流的确认和状态转换。

**原型映射**

`demo.html` 的右侧辅助追问保持独立；追问发送后只解释当前来源，不切换中间主流内容。

### ITER-005：复盘和重练能不能只靠消息文本

**问题**

如果复盘、错因、重练题都靠解析消息文本，将难以稳定落地，也无法可靠做会话锁定、薄弱点统计和历史复盘。

**发散想法**

- `MVP 保留`：保留 `messages` 作为 UI 回放记录。
- `MVP 保留`：新增 `exercise_attempts`、`grading_results`、`error_tag_events` 三类轻量索引。
- `后续候选`：增加 `learning_goals`、`review_snapshots` 支持长期学习计划。
- `暂不做`：不建立复杂学习知识图谱。
- `已砍掉`：不依赖自然语言消息临时解析生成复盘。

**MVP 决策**

消息流是体验记录，结构化索引是业务记录。二者并存，避免把聊天记录误当数据库。

**原型映射**

`workflows.html` action matrix 说明复盘从结构化记录读取；`demo.html` 只展示用户可见结果，不暴露表结构。

### ITER-006：会话锁定会不会过重，影响历史复盘

**问题**

防止练习时回看答案是合理的，但如果完全禁止历史，会损害“完整历史复盘”的产品特色。

**发散想法**

- `MVP 保留`：历史会话仍可见。
- `MVP 收敛`：当前题未完成时，只隐藏答案、参考答案和批改详情。
- `后续候选`：根据题型或考试模式提供更严格的锁定。
- `暂不做`：不做复杂权限层级和计时考试模式。
- `已砍掉`：不完全禁止打开历史会话。

**MVP 决策**

采用温和锁定：能看历史摘要，不能看可能泄题的答案细节。练习完成后恢复完整复盘。

**原型映射**

`demo.html` 左侧历史展示受限状态；`workflows.html` 展示 `conversation-lock` 边界流程。

### ITER-007：账号、注册、保存是否应该独立成页面

**问题**

注册登录是必要能力，也可以算作 Widget，但真实认证通常需要系统路由、表单校验和安全流程。若全塞进 AI 对话，会有安全和实现风险。

**发散想法**

- `MVP 保留`：`account-gate` 是对话内账号入口 Widget，支持登录、注册、稍后保存。
- `MVP 收敛`：Widget 只收集意图和引导，真实鉴权由系统组件或路由承接。
- `后续候选`：支持 magic link、第三方登录、家长账号绑定。
- `暂不做`：不在 V1 原型里实现真实表单校验和真实账号创建。
- `已砍掉`：不让 AI 直接处理密码、token 或权限判断。

**MVP 决策**

注册登录算作 Widget 入口，但不是 AI 自由生成的表单。安全边界归系统。

**原型映射**

`widgets.html` 保留 `account-gate`；按钮说明为账号 action；`demo.html` 只展示保存学习进度的轻量入口。

### ITER-008：自定义场景、跳过和提示是否会破坏练习闭环

**问题**

自定义场景、跳过、要提示都是自然需求，但如果不受状态机约束，会导致用户随时绕过答题或生成无法评分的内容。

**发散想法**

- `MVP 保留`：自定义场景只在 `scene_selecting` 生效，系统确认后进入练习。
- `MVP 收敛`：提示只给方向，不直接给完整答案。
- `MVP 收敛`：跳过仍生成一次 attempt，标记为 skipped，便于复盘。
- `后续候选`：根据用户连续跳过次数调整难度。
- `已砍掉`：不允许练习中直接改写当前题为另一个题而不记录状态。

**MVP 决策**

所有看似自由的学习动作都要落到结构化状态：选择、提交、提示、跳过都必须可记录、可复盘。

**原型映射**

`workflows.html` 的 action matrix 明确每个状态允许动作；`demo.html` 的自定义场景入口回到自然输入，不直接伪装为已开始练习。

### ITER-009：工作流展示页应该解释候选分支，但不能让人误会已进入 MVP

**问题**

用户希望需求像树一样生长，但真实 demo 又必须像上线页面。若把候选分支做进 demo，会误导实现范围；若完全不展示候选，又丢失产品思考过程。

**发散想法**

- `MVP 保留`：`demo.html` 只展示当前选定路径。
- `MVP 保留`：`widgets.html` 只展示当前 V1 Widget 契约。
- `MVP 收敛`：`workflows.html` 展示主线、候选、暂不做和砍掉项，并明确状态。
- `后续候选`：增加产品决策看板或版本路线图。
- `已砍掉`：不在真实产品演示中加入原型框架、调试面板或内部状态开关。

**MVP 决策**

把“真实体验”和“需求解释”分开。demo 负责像产品，workflows 负责讲清楚产品树。

**原型映射**

本轮在 `workflows.html` 增加发散候选与取舍区；不改变 `demo.html` 的真实产品感。

## 4. 持续迭代协议

后续每次发现新问题，按以下格式追加：

```text
### ITER-XXX：问题标题

问题：
当前需求或原型哪里不清晰、不完整或可能跑偏。

发散想法：
- MVP 保留：
- MVP 收敛：
- 后续候选：
- 暂不做：
- 已砍掉：

MVP 决策：
当前版本选择什么，为什么。

原型映射：
demo / widgets / workflows 分别如何展示，哪些不展示。
```

## 5. 当前约束

- MVP 路径不能依赖任何 `后续候选` 功能。
- `暂不做` 和 `已砍掉` 不能出现在 `demo.html` 主路径。
- 所有进入 MVP 的按钮、Widget action、状态转换都必须能映射到 `08` 契约。
- 发散记录可以开放生长，但实现基准必须保持可落地。
